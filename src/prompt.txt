Master Roadmap: Steam Achievement Tracker API
Part 1: Architecture & Tech Stack
To impress hiring managers, we will use industry-standard tools and avoid starter defaults.
IDE: IntelliJ IDEA (JetBrains).
Framework: Spring Boot 3.5.x with Java 21.
Database: PostgreSQL (Use H2 in-memory database strictly for initial local testing, migrating to Postgres for production).
Data Access: Spring Data JPA / Hibernate.
Security: Spring Security with Session-Based Authentication (Form Login) for stateful session management.
External Data Providers: * Steam Web API: The core engine. Used for fetching user owned games, static achievement schemas, and user unlock timestamps.
RAWG.io / IGDB: For data enrichment (fetching high-quality cover art, since Steam's default grid images can be low-res).
Part 2: Database Design & Entity Relationships
User Entity: Stores account details (UUID, username, email, BCrypt-hashed password) and their required Steam ID.
Game Entity: Stores game metadata (ID = Steam AppID, title, cover_art_url).
Achievement Entity (The Blueprint): Stores the static definition of an achievement (ID, Steam apiName, displayName, description, iconUrl).
UnlockedAchievement (The Event): An intermediate entity tracking exactly when a specific user earned a specific achievement.
Part 3: Step-by-Step Implementation Plan
Step 1: Account Creation & Linking Steam
Action: The user registers, logs in via the default Spring login form, and saves their public Steam ID.
How it works: You will build a registration endpoint. Once registered, the user logs in. Spring Boot's default formLogin() handles the credential check, establishes a secure server-side session,
and issues a JSESSIONID cookie to the client. Subsequent requests are authenticated automatically via this cookie.
Step 2: Fetching the Game Library (Data Ingestion)
Action: The user clicks "Sync Steam Library."
How it works: Using Spring's modern RestClient, your backend calls the Steam API's GetOwnedGames endpoint. It extracts the AppIDs, checks if those games exist in your Game table, and saves any new ones so your system knows they exist.
Resume Skill: Proves you can consume and parse massive third-party JSON payloads efficiently.
Step 3: Fetching Game Metadata (Data Enrichment)
Action: The background system makes the games look pretty.
How it works: Your backend takes the AppIDs and queries RAWG.io/IGDB to fetch high-res cover art, updating the Game entity. Data Transfer Objects (DTOs) package this cleanly for the frontend.
Resume Skill: Shows orchestration of multiple external services and API rate-limit handling.
Step 4: Syncing Achievements (Core Business Logic)
Action: The user clicks "Sync Progress" on a specific game (e.g., CS:GO).
How it works: 1. Spring Boot calls Steam's GetSchemaForGame to ensure all possible achievements for CS:GO exist in your Achievement table. 2. It calls Steam's GetPlayerAchievements to see which ones the user actually unlocked. 3. It iterates through the response, converting Steam's Unix timestamps into Java LocalDateTime, and saves records to the UnlockedAchievement table.
Resume Skill: Validates understanding of complex relational database design (bridging static vs. dynamic data) and historical data ingestion.
Part 4: The "Resume Booster" Polish
This is where the project goes from "tutorial follower" to "hirable engineer."
Global Exception Handling: Use @ControllerAdvice to catch errors gracefully (e.g., returning a clean JSON error if a user's Steam Profile is set to "Private").
Interactive Documentation: Integrate springdoc-openapi to auto-generate a Swagger UI webpage so recruiters can test your API visually.
Deployment: Containerize the Spring Boot app with a Dockerfile and deploy the Postgres database and app to a cloud provider like Render, Railway, or Fly.io.
